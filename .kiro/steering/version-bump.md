# กฎการอัพเดทเวอร์ชันโปรแกรม

ทุกครั้งที่แก้ไขโค้ดของโปรเจกต์นี้ **ต้องเพิ่มเลขรุ่นย่อย (patch version)** ในไฟล์ `frontend/src/lib/version.ts` เสมอ เพื่อให้ผู้ใช้เห็นได้ว่ามีการอัพเดท

## ไฟล์ที่ต้องแก้

`frontend/src/lib/version.ts`

```typescript
export const APP_VERSION = 'X.Y.Z'
export const APP_BUILD_DATE = 'YYYY-MM-DD'
```

## วิธีเพิ่มเวอร์ชัน (Semantic Versioning: MAJOR.MINOR.PATCH)

- **PATCH** (Z) เพิ่ม +1 — แก้ทุกครั้งที่มีการเปลี่ยนแปลงโค้ด เช่น bug fix, ปรับ UI เล็กน้อย, refactor
  - ตัวอย่าง: `1.5.0` → `1.5.1` → `1.5.2`
- **MINOR** (Y) เพิ่ม +1 และรีเซ็ต PATCH เป็น 0 — เมื่อเพิ่มฟีเจอร์ใหม่ที่ไม่ทำให้ของเดิมพัง
  - ตัวอย่าง: `1.5.3` → `1.6.0`
- **MAJOR** (X) เพิ่ม +1 และรีเซ็ต MINOR/PATCH เป็น 0 — เมื่อเปลี่ยนแปลงใหญ่ที่ทำให้ของเดิมใช้ไม่ได้ (breaking change)
  - ตัวอย่าง: `1.6.5` → `2.0.0`

## สิ่งที่ต้องทำทุกครั้งที่แก้โค้ด

1. แก้ไขโค้ดตามที่ผู้ใช้ขอให้เสร็จก่อน
2. อัพเดท `APP_VERSION` ใน `frontend/src/lib/version.ts` (เพิ่ม PATCH เป็นค่าเริ่มต้น)
3. อัพเดท `APP_BUILD_DATE` เป็นวันที่ปัจจุบัน (รูปแบบ `YYYY-MM-DD`)
4. อัพเดท `CACHE_NAME` ใน `frontend/public/sw.js` ให้ตรงกับเวอร์ชัน เช่น `yaduang-pos-v1.5.2` — สำคัญมาก ไม่งั้น PWA จะใช้ cache เก่า ผู้ใช้จะไม่เห็นการเปลี่ยนแปลง
5. แจ้งผู้ใช้ว่าอัพเดทเป็นเวอร์ชันอะไร

## ข้อยกเว้น (ไม่ต้องเพิ่มเวอร์ชัน)

- แก้ไฟล์ที่ไม่กระทบโปรแกรมจริง เช่น `.md`, comment, ไฟล์ใน `.kiro/`, `.git/`
- แก้แค่ฝั่ง backend อย่างเดียวที่ไม่กระทบ user — แต่ถ้าไม่แน่ใจ ให้เพิ่ม PATCH ไว้ก่อน
